## Risk Register Template

Track project risks in a simple spreadsheet or table:

| ID | Risk Description | Likelihood | Impact | Risk Score | Mitigation Strategy | Owner | Status |
|---|---|---|---|---|---|---|---|
| R1 | Vendor delivers integration late | High | High | Critical | Build 2-week buffer; identify backup vendor | Galen | Open |
| R2 | Key team member on PTO during migration | Medium | High | High | Cross-train backup person before PTO | Galen | Mitigated |
| R3 | Users resist new system | High | Medium | High | Early involvement; training; executive sponsor comms | Galen | Open |
| R4 | Data migration has quality issues | Medium | Medium | Medium | Validate sample data first; automate validation checks | IT Team | Open |
| R5 | Budget overrun on licenses | Low | Medium | Low | Get firm quote upfront; negotiate multi-year discount | Galen | Closed |

**Risk Score Guide:**
- **Likelihood √ó Impact** ‚Üí Critical / High / Medium / Low
- Review weekly in sprint planning
- Close risks that no longer apply or have been fully mitigated

## Project Checklist Templates

### System Migration Checklist

```
PRE-MIGRATION:
‚ñ° Backup all data from source system
‚ñ° Verify backup can be restored
‚ñ° Document current system configuration
‚ñ° Map data fields from old to new system
‚ñ° Identify data that needs cleanup before migration
‚ñ° Test migration with sample data set
‚ñ° Create rollback plan and test it
‚ñ° Schedule migration window with stakeholders
‚ñ° Notify end users of downtime and timeline
‚ñ° Prepare support team for post-migration issues

MIGRATION DAY:
‚ñ° Confirm final backup is complete
‚ñ° Begin migration per documented procedure
‚ñ° Validate data in new system (spot-check records)
‚ñ° Run automated validation scripts
‚ñ° Test critical business processes
‚ñ° Verify integrations with other systems
‚ñ° Get sign-off from key users

POST-MIGRATION:
‚ñ° Monitor system performance for 48 hours
‚ñ° Track and resolve user-reported issues
‚ñ° Verify all scheduled jobs/automations are running
‚ñ° Decommission old system (after confirmation period)
‚ñ° Update documentation and SOPs
‚ñ° Conduct lessons-learned review
‚ñ° Archive migration records
```

### New Software Rollout Checklist

```
PLANNING:
‚ñ° Define business requirements and success criteria
‚ñ° Evaluate vendors (minimum 3)
‚ñ° Complete security review / vendor questionnaire
‚ñ° Negotiate contract and pricing
‚ñ° Get budget approval
‚ñ° Plan integration with existing systems

SETUP:
‚ñ° Provision accounts and licenses
‚ñ° Configure SSO / authentication
‚ñ° Set up roles and permissions
‚ñ° Configure integrations (API, data sync)
‚ñ° Set up backup and monitoring
‚ñ° Create test environment

TESTING:
‚ñ° Internal IT testing (functionality)
‚ñ° Security testing (penetration, access controls)
‚ñ° Integration testing (data flows correctly)
‚ñ° User acceptance testing (representative users)
‚ñ° Performance testing (acceptable load times)
‚ñ° Document and resolve all test failures

ROLLOUT:
‚ñ° Create user training materials
‚ñ° Conduct training sessions
‚ñ° Communicate launch timeline to all users
‚ñ° Staged rollout (pilot group ‚Üí department ‚Üí all)
‚ñ° Dedicated support channel for first 2 weeks
‚ñ° Daily monitoring of adoption metrics

POST-LAUNCH:
‚ñ° Gather user feedback (week 1 and week 4)
‚ñ° Address top issues
‚ñ° Optimize configuration based on usage patterns
‚ñ° Update SOPs and documentation
‚ñ° Close project and conduct retrospective
```
